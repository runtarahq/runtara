# Connections as workflow-as-agent capability inputs — design

_Produced via multi-agent mapping + synthesis, verified against the feature branch (2026-07-15)._

## Decisions from review (2026-07-15) — these REFINE the design below

1. **Binding model = named logical slots** (§2.i/ii as written). A workflow declares an integration-typed `connectionSchema` of named slots; steps reference a slot; the caller binds each slot to one of its own connection ids via the `connections` envelope. Multiple same-integration steps may share or use distinct slots — the author decides.
2. **A connection binding is a RESOLVABLE VALUE, not only a slot name.** In addition to `{slot: "<name>"}`, a step's connection must accept a literal `{connectionId: "conn-123"}` (same-tenant, back-compat) AND a runtime reference `{valueType: "reference", value: "data.chosenConnectionId"}`. Rationale from the owner: (a) **connection ROTATION** — swap which concrete id a slot/step uses by changing the bound VALUE, never the workflow structure; (b) **pass connection ids INSIDE the workflow** — a step can resolve a connection id from the run's own data (e.g. pick the connection per record being processed). So the connection field on a step is a small tagged union that resolves to ONE concrete connection id at runtime; the named slot is the external interface, the value is the flexible binding. This unifies "reference a slot", "pin a literal", and "compute an id from data".
3. **Rate limiting is per concrete connection id — CONFIRMED already true**, so rotation/dynamic selection is correct for free. The proxy keys the limiter on the resolved id: `internal_proxy.rs:388` `check_rate_limit(connection_id, &conn.rate_limit_config)`; the rate-limit reporting surface is entirely `connectionId`-keyed. No change needed — but any future slot-level rate-limit UI must still resolve to the concrete id, not the slot.
4. **Status: DESIGN ONLY** — not implemented. First PR remains the DSL connection-reference / `connectionSchema` work (§5), valuable standalone (reusable workflows + env promotion), landing in lockstep with slice c's `meta.json` connection-slots descriptor.

# Connections as capability inputs — design answer

## 1. The two sub-problems, precisely

**(a) Connections are pinned literals, not bindable references.** A step carries `connection_id: Option<String>` — a bare `String`, not a `MappingValue` — on both `AgentStep` (`schema_types.rs:453-455`) and `AiAgentStep` (`:1237`), each on a `deny_unknown_fields` struct. It flows verbatim into `DirectAgentManifest.connection_id` (`manifest.rs:404`, cloned at `:980/1136/1186/1216/1248`), is **baked into linear memory as a static data segment** (`static_data.rs:391-394`, `DirectDataSegment::new(offset, connection_id.as_bytes())`; accessor `:319`), and is lowered by *reading that segment* — `emit_agent_connection_args` at `agent_invoke.rs:70` reads `static_data.agent_connection_id(agent_id)` and writes it id-only. Data mappings, by contrast, *are* parameterizable: `MappingValue` (`schema_types.rs:1451-1465`) is a `valueType`-tagged union with a `Reference` arm (`ReferenceValue :1488`) resolved at runtime. Connections have no such form. There is no reference, template, or variable shape for a connection, and no per-workflow "connection requirement" concept anywhere (grep-confirmed; the one `get_agent_connection_schema_handler` at `operators.rs:124` is a 501 stub). **Confirmed: connections are not references.**

**(b) The agent signature carries exactly one connection; a workflow needs many.** The shared WIT is `invoke(capability-id, input: list<u8>, connection: option<connection-info>)` (`runtara-agent.wit:38-42`), one optional `connection-info` (`:16-22`), mirrored at the workflow boundary (`runtara-abi.wit:18-22`). A workflow that touches N distinct connections (HubSpot + S3 + Postgres…) has one slot to receive them. Worse, for a workflow-as-agent that one slot is **discarded today**: `WorkflowAbi::AgentCapabilities` "accepts but ignores connection in v1" (`component.rs:82-84`), and the lowering literally loads `input` from `[params+8]` and skips `connection` at offset `@16` (`core_module.rs:551-552`). So even the single connection goes nowhere; inner steps keep using their own compile-time-pinned ids. **Confirmed: one-slot signature is structurally insufficient, and currently inert.**

These are independent gaps. (a) is about *authoring* (how a step names a connection); (b) is about *transport* (how a caller supplies N of them across the invoke boundary).

## 2. Recommended model — confirm/refute each part

The hypothesis holds, with two concrete corrections. Evaluated part by part:

**(i) Declare a CONNECTION SCHEMA — named, integration-typed slots, beside `inputSchema`. CONFIRM.** The natural home is `ExecutionGraph` (`schema_types.rs:71`) beside `input_schema` (`:100`) and `output_schema` (`:105`): add `connection_schema: HashMap<String, ConnectionSlot>` where `ConnectionSlot = { integration_id, required, description }`. This is the design-time declaration of what the workflow needs. It must **not** be baked into a static segment (unlike `direct_core_variables_json`, `static_data.rs:51`) — it is caller-supplied, so it rides the invocation. Note this is genuinely new; the only existing "connection schema" is `ConnectionTypeMeta` (`agent_meta.rs:1004-1024`), which describes how to *create* a connection of an agent type — a different thing. The 501 stub even reserves the endpoint name.

**(ii) Steps REFERENCE a slot instead of a pinned id. CONFIRM, with a correction on representation.** `connection_id` must become bindable. Two options: reuse `MappingValue::Reference` pointing at `connections.<slot>`, or add a dedicated `ConnectionRef` form. **Recommend a dedicated `connectionRef` sibling field over overloading `connection_id` into a `MappingValue`.** Reasons: (1) credentials are deliberately out-of-band and opaque — conflating a credential binding with a data reference invites someone to point a data mapping at a connection or vice-versa; (2) it keeps `connection_id` as the same-tenant literal (back-compat, no migration of existing pinned ids); (3) validation stays legible. The resolution point moves: `emit_agent_connection_args` (`agent_invoke.rs:65-104`) and `agent_connection_input` (`direct_json.rs:2248-2276`) stop reading the static segment for *bound* steps and instead resolve the slot from the runtime `source`; `collect_static_agent_data` (`static_data.rs:391`) bakes only *still-literal* ids.

**(iii) Slots are BOUND to concrete ids by the caller at invocation. CONFIRM — and this is forced, not optional.** The cross-tenant reality (§3) means a baked id is useless in the consumer tenant, so binding *must* happen at invoke time. This is exactly why the emitter's static-segment read must give way to runtime resolution for bound steps.

**(iv) The binding MAP rides in the INPUT envelope as opaque ids; the single `connection` param is reserved for the single-connection native case / a primary slot. CONFIRM — this is the right carrier — but two subtleties must be handled.** The mechanism: extend the canonical envelope `{data, variables}` → `{data, variables, connections}` where `connections = { slot: { connection_id } }`. `build_source` (`direct_json.rs:3160-3235`) already assembles a flat namespace map (`data`, `variables`, `steps`, `workflow.inputs`, `loop`, `item`); insert a `connections` namespace at ~`:3215`. Reference resolution then works **for free** — the first path segment is a plain top-level key (`lookup_segments_detailed :5732`, `path_to_segments :5687`), so `connections.<slot>.connection_id` resolves with no special-casing. **Zero WIT change, N connections over the existing `input: list<u8>` param.** The two subtleties the maps surface:

- **The parent produces a *flat* child input, not an envelope.** `emit_agent_plan` (`agent.rs:94-108`) applies the step input-mapping and hands the child a flat capability-input object. `build_source` treats an input with no `data` key *as* `data` (`:3172-3177`), so a top-level `connections` key would be swallowed into `data.connections`. Fix: for a workflow-agent child, the parent Agent step must wrap the child input as `{data, variables, connections}`, and `build_source` must lift `connections` out. This is the load-bearing wiring change.
- **The single `connection` param stays, but only for the native single-connection case.** A plain native agent (one connection) keeps using the WIT `connection` slot exactly as today. For a workflow-agent it remains ignored, or is optionally folded into the map as a designated `primary` slot. Do **not** try to push the map through that slot — see §4.

**Net recommendation:** connectionSchema on the graph → `connectionRef` on steps → runtime resolution from a `connections` namespace lifted out of the input envelope. The single WIT `connection` param is left as the native-agent path.

## 3. Credential boundary — why a map of ids is safe, and what breaks cross-tenant

**Carrying a map of ids through `input` is secure, verified against the maps.** A connection is an opaque per-tenant *id*; it becomes credentials only server-side, at outbound-call time, keyed by `(connection_id, tenant_id)`. The single injection point is the proxy: `internal_proxy.rs execute_proxy_request` calls `get_with_parameters(connection_id, tenant_id)` (`:277`), `resolve_connection_auth` injects Authorization/OAuth/SigV4 (`:304-318`), and `pin_url_to_base` fail-closes the host (`:334-376`). The guest only ever forwards the id via `X-Runtara-Connection-Id`, and `runtara-http` **strips all `x-runtara-*` headers** and forwards tenant via `X-Org-Id`/`RUNTARA_TENANT_ID` (`lib.rs:189-263`). The compiled emitter already writes connection args **id-only** — integration empty, `parameters = {}` (`agent_invoke.rs:65-104`, `static_data.rs:31`). So a `connections` map of ids carries nothing the guest couldn't already see; it never carries secrets. **Confirmed safe — provided the map carries only ids.**

**One divergence to close:** the *test/dispatch* path materializes real secret params into the guest — `dispatcher.rs:253` does `parameters: to_string(&c.parameters)`, unlike the emitter's `{}`. The workflow-as-agent path must mirror the emitter (`parameters = {}`), not the dispatcher. (Caveat: any agent that actually reads `_connection.parameters` — HubSpot does at `lib.rs:3244-3262` — would need to keep routing via the proxy; this is the argument for eventually dropping `parameters` from the boundary entirely.)

**What breaks if a bound id belongs to a different tenant (the published case):** nothing silently — it fail-closes. `get_with_parameters` is `WHERE id=$1 AND tenant_id=$2` (`repository/connections.rs:798`). When consumer tenant T runs a workflow-agent published by tenant P, the inner steps resolve under **T's** proxy (`X-Org-Id` = T, because in-guest composition runs the whole artifact in one tenant env — this is precisely why the plan chose composition over host-dispatch). If the binding map contains an id that T does not own, `get_with_parameters` returns nothing and the outbound call fails — no cross-tenant credential leak is possible, because P's ids simply don't resolve under T.

**Who validates ownership at bind time:** the **caller (consumer) tenant's save-time validation.** `extract_connection_ids` (`connection_validation.rs:24-59`) already walks a workflow — recursing Split/While — and checks every referenced id against the tenant's own connection set (missing → E026, `validation.rs:3285`). Extend it so that when a parent binds a child capability's slots, each bound id is validated as owned by the *parent's* tenant, exactly as literal ids are today. A tenant can only bind ids it owns; ownership is enforced where it already is.

## 4. Alternatives and why not

- **Widen the WIT `connection` to `list<connection-info>` (or a map).** Rejected — blast radius. `runtara-agent.wit` is the *shared* contract for every native agent; the typed host sig `(String, Vec<u8>, Option<ConnectionInfo>)` (`dispatcher.rs:295-299`), every agent's generated bindings, and the 17-flat-param lowering (`core_module.rs:546`) all move. You'd break every agent to serve the workflow case. The input-envelope carrier gets N connections with zero WIT/ABI change and dovetails with existing input mappings.

- **Bake the publisher's connection ids into the compiled child.** Rejected — this is literally today's behavior (`static_data.rs:391`) and the thing that breaks cross-tenant: `get_with_parameters` is tenant-scoped, so a publisher id is meaningless to a consumer. Keep literal baking *only* as a same-tenant authoring convenience (the `connection_id` literal path), never for published capabilities.

- **A separate `bind-connections` host call.** Rejected for v1 — it reintroduces a guest→host runtime boundary that slice d deliberately removed (a pure workflow-agent imports zero runtime), needs a new host resolver, and can't be done purely in-band. It only earns its keep if we also move to host-side dispatch (architecture B) for durable/suspending children — which is out of v1 scope. Named as the escape hatch, not the path.

The tradeoff summary: the input-envelope map is the least-invasive, composition-safe option; its cost is that `build_source`/parent-wrap must learn about a `connections` namespace, and `connection_id` gains a bindable sibling. That cost is local and additive.

## 5. Phasing

**Minimal first step — connection references + connectionSchema in the DSL, valuable on its own (do this regardless of workflow-as-agent):**

1. `ExecutionGraph.connection_schema: HashMap<String, ConnectionSlot>` beside `input_schema` (`schema_types.rs:100`).
2. `connectionRef` sibling on `AgentStep`/`AiAgentStep` (`schema_types.rs:455/1237`); `connection_id` literal stays for back-compat, no migration.
3. `build_source` lifts a `connections` namespace into `source` (`direct_json.rs:~3215`); references resolve for free.
4. `emit_agent_connection_args` (`agent_invoke.rs:70`) + `agent_connection_input` (`direct_json.rs:2248`) resolve bound steps from `source`; `collect_static_agent_data` (`static_data.rs:391`) bakes only literals.
5. Validation: extend `extract_connection_ids` (`connection_validation.rs:24`) and E026/E112 to accept "satisfied by a declared slot," and to validate literal/bound ids as tenant-owned.

This alone unlocks **reusable workflows** (bind one logical `hubspot` slot to different connections) and **environment promotion** (dev→prod: same workflow, different connection ids supplied per environment in the `connections` envelope). It needs *no* workflow-as-agent machinery — a top-level run just passes `connections` in the input envelope. Ship and verify it standalone.

**What workflow-as-agent invoke wiring adds on top (this is the plan's deferred slice e):**

- Parent Agent step targeting a workflow-agent child wraps child input as `{data, variables, connections}` (change in `emit_agent_plan`/`agent.rs`) and lets the author bind the child's declared slots to the parent's own refs/literals.
- The child's `connectionSchema` is advertised so parents (and validators) can see required slots.
- The single WIT `connection` param stays the native single-connection path; no WIT change; the ignored slot at `core_module.rs:551` optionally becomes a "primary" slot merged into the map.

**Interactions with the slug plan and slices b/c:**

- **Slice c is the coordination point.** Slice c synthesizes `runtara_agent_<id>.meta.json` = `AgentInfo` with a `CapabilityInfo` whose `inputs` = the workflow input schema (`agent_meta.rs:308`). The connection requirements belong right here: add a `connectionSlots` descriptor beside `inputs`, derived from `connection_schema` (or auto-derived via `extract_connection_ids`). That is what lets the step-picker, `validate_graph`, and E124 closure see and check required slots when a parent binds a child. **The connection-schema work should land in lockstep with slice c's meta synthesis.**
- **Slice b (staging/composition) is unaffected** — because the map rides in-band, there is no wac/WIT change, so `validate:true` composition stays clean. This is a feature of choosing the envelope carrier.
- **Slug plan is orthogonal but adjacent.** Slug = the capability *id* (`agent-<slug>`, replacing the `CAPABILITIES_EXPORT_AGENT_ID = "workflow-agent"` const at `component.rs:245`); connections = the capability *connection inputs*. Both are metadata the consumer needs, both surface through slice c. The slug plan's cross-tenant note (capability ids get publisher-qualified at the marketplace layer, local `workflows.slug` stays per-tenant) is exactly the right shape for connections too: **slots are logical names local to the capability; the consumer binds them to its own ids — no publisher id ever crosses the tenant boundary.**

Dependency-wise: the DSL connection-reference work slots in as its own PR train (mirrors the slug plan's PR ordering) and *feeds* slice c; slice e becomes "wire the parent-wrap + advertise slots" rather than a from-scratch design.

## 6. Open questions the maps could not resolve

1. **Tenant propagation into a composed child.** In-guest composition *should* run the whole artifact under the parent's tenant env (so inner `X-Org-Id`/`RUNTARA_TENANT_ID` = consumer tenant, `lib.rs:247-263`), which is what makes cross-tenant binding coherent. But the maps flag this as unverified for nested workflow components — confirm the consumer tenant actually reaches the child's inner agent calls before relying on it.
2. **Binding key granularity.** Is a child slot bound by logical slot name (recommended), by `integration_id`, or by original inner `connection_id`? And how does *one* caller-supplied connection map to *many* inner steps of the same integration? The maps recommend logical named slots but don't settle the many-to-one mapping.
3. **Should `ConnectionInfo.parameters` be dropped from the boundary entirely?** The emitter sends `{}`, the dispatcher sends real secrets (`dispatcher.rs:253`). Id-only-everywhere is cleaner, but agents that read `_connection.parameters` (HubSpot, `lib.rs:3244-3262`) would break — needs an audit of which agents actually consume params vs. route via the proxy.
4. **Non-HTTP native agents.** SFTP carries an opaque `_connection` and may resolve creds through a different host path than the HTTP proxy. Whether that path also honors consumer-tenant remapping is unconfirmed.
5. **Literal-vs-ref union validation.** Exact behavior of E026/E112 (`validation.rs:682/1068/3285`) when a step is "satisfied by a caller-supplied binding" rather than a present literal — needs a decision on required-ness and default slots.
6. **Standalone `runtara-compile --child`.** The CLI has no DB; resolving a child's declared slots (and slug) without the `workflow_dependencies` join is an open gap the slug plan already flags — connections inherit the same problem.
