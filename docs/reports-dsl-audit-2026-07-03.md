# Reports DSL audit — inconsistencies and gaps

Date: 2026-07-03 (completed in two passes). Method: 11 parallel analyzers over distinct DSL dimensions (core types, expression languages, format/templates, validation, edit ops, renderers, provider parity, MCP authoring schema, frontend parity, API/lifecycle, test corpus), cross-analyzer dedupe, one adversarial verifier per finding instructed to refute it against the code, then a completeness-critic round (preview parity, mutation concurrency, workflow-button contract, pagination) with its own find+verify pass.

Stats: 141 raw findings -> 98 deduped (F01-F98) + 34 critic-round findings (X0-X33). **131 confirmed, 1 refuted (F87).** Every confirmed finding below survived an adversarial verification pass with file:line evidence. Note: X findings were deduped only within their cluster, so several restate the same root cause from different angles; duplicates of F findings are marked.

Layers audited: crates/runtara-report-dsl, crates/runtara-server (services/reports + providers + renderers + query_plan, handlers/dto/repositories, MCP tools), runtara-object-store SQL condition layer, frontend features/reports + WASM bindings, server test corpus.

## Verified findings

### Condition & expression semantics diverge across evaluation layers

**F01 [high/inconsistency] CONTAINS has three semantics: case-insensitive ILIKE in SQL pushdown, case-sensitive string-only in server in-memory eval, string+array membership in client WASM**

- Primary: `crates/runtara-server/src/api/services/reports/query_plan.rs:296`
- Impact: Same condition/search box returns different rows per source kind; adding a join to an object_model block silently flips search from case-insensitive to case-sensitive and drops SQL-fetched rows in the in-memory recheck.
- Evidence: SQL pushdown: crates/runtara-object-store/src/sql/condition.rs:624-626 (`format!("%{}%", value)`; `"\"{}\"::text ILIKE ${}::text"`). Server in-memory: crates/runtara-server/src/api/services/reports/query_plan.rs:296-313 (`left.as_str().is_some_and(|haystack| haystack.contains(needle))`). Client WASM: crates/runtara-report-dsl/src/row_condition.rs:111-116 (Op::Contains dispatch) and 323-329 (string substring + array membership via values_equal). Joined-block double evaluation: crates/runtara-server/src/api/services/reports.rs:2856-2857 (primary_pushdown_condition → SQL) then 2899-2909 (full condition re-checked in-memory via condition_matches_row); pushdown keeps non-alias conditions per query_plan.rs:190-194. Search always emits CONTAINS: reports.rs:6009-6017.

**F02 [high/inconsistency] Null-field comparison semantics diverge across all three condition evaluators: server in-memory matches null for GT/GTE/NE, SQL pushdown excludes null for all comparisons, client WASM excludes GT/GTE but matches NE**

- Primary: `crates/runtara-server/src/api/services/reports/query_plan.rs:427`
- Impact: GT/GTE/NE filters on system/workflow_runtime sources (or post-join rechecks) include rows with missing data that the identical condition on plain object-model sources excludes; FE optimistic eval disagrees with the server.
- Evidence: crates/runtara-server/src/api/services/reports/query_plan.rs:427 `(Value::Null, _) => Some(Ordering::Greater)` in compare_values (424-434), consumed by compare_binary (345-350) for the GT (267-269), GTE (270-272), NE (264-266) arms — null/missing field matches. Callers: crates/runtara-server/src/api/services/reports/providers/system.rs:75,172; providers/workflow_runtime.rs:79,674; reports.rs:2903,7816 (join post-filter). SQL pushdown: crates/runtara-object-store/src/sql/condition.rs:592-595 `\"{field}\"::{cast} {op} ${n}::{cast}` — NULL excluded via SQL three-valued logic for GT/GTE/NE/LT/LTE. Client WASM: crates/runtara-report-dsl/src/row_condition.rs:258-275 GT/GTE/LT/LTE require both operands numeric-or-string else false (null excluded), but line 257 `Op::Ne => !values_equal(left, right)` matches null vs non-null; wired to FE via frontend/src/features/reports/utils.ts:225 (evaluateRowCondition WASM export).

## Refuted

**F87: ObjectModelProvider::field_is_known is unconditionally true, contradicting the trait doc's claim of schema-aware checking — provider-interface consumers accept unknown fields for object_model**

- Why refuted: The literal code claim is true (ObjectModelProvider::field_is_known returns true unconditionally at object_model.rs:100-102), but the impact claim is refuted by the surrounding call graph. The only consumer of field_is_known/markdown_field_known in the entire runtara-server crate is reports.rs:1370-1372, and that call is gated at reports.rs:1359-1362 to `matches!(block.source.kind, ReportSourceKind::WorkflowRuntime | ReportSourceKind::System)` — object_model blocks never reach the trait-routed check. Instead, object_model blocks fall through to a schema-aware branch: the tenant schema is dynamically loaded (reports.rs:1382-1395), a join-alias-aware `is_known_field` closure is built (reports.rs:1429-1437), and markdown placeholders are validated against it at reports.rs:1680-1685, rejecting unknown fields at save time with UNKNOWN_MARKDOWN_SOURCE_FIELD (reports.rs:4573-4582). So unknown object_model markdown fields do NOT 'silently pass validation and surface as render-time nulls' — they are rejected by validate_report, exactly like (in fact more precisely than) the sibling providers. The split is also documented as intentional: object_model.rs:4-7 states object-model definitions are validated by the generic validate_report_definition path against the dynamically-loaded schema, with the schema-aware validator moving into the provider in Phase 5. What remains is only a doc nit — the trait doc (providers.rs:105-107) describes the stub optimistically, and the stub would matter only if a future consumer routed object_model through the trait default — which is a latent code-quality observation, not an observable inconsistency or validation gap per the audit's definitions. This matches refutation path (1): a validation rule already rejects the problematic definition.
